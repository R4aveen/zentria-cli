# ₊⊹ ✴︎ Zentria CLI — Release v1.1.6

> **Fecha:** 12 de marzo de 2026  
> **Branch:** `main`  
> **Estado:** En período de pruebas — errores conocidos en impresión de etiquetas

---

## Resumen

Esta versión resuelve el mayor bloqueador para la distribución interna: **Windows Defender y SmartScreen**. Se implementa un pipeline completo de firma digital con certificado autofirmado embebido en el ejecutable, un componente de instalación automática con pantalla de progreso, y un esquema de distribución mediante ZIP + launcher `.bat` que no activa SmartScreen. Además se normaliza el versionamiento del ZIP con el número de versión del proyecto y se establece un workflow de CI/CD con GitHub Actions.

---

## Cambios Incluidos

### ✦ 1. Firma digital del ejecutable  
**Commits:** `b8c04b0`, `88bcbdd`

El ejecutable ahora está **firmado digitalmente** con un certificado autofirmado `ZentriaCLI` generado en el almacén `Cert:\CurrentUser\My` de Windows.

**Proceso en `build-exe.mjs`:**
1. Se crea (si no existe) un certificado `CodeSigningCert` con `New-SelfSignedCertificate` — validez 5 años.
2. Se exporta a `build/ZentriaCertificado.cer`.
3. Tras inyectar el blob SEA y aplicar metadatos con resedit, se firma el `.exe` con `Set-AuthenticodeSignature`.

```powershell
# Verificar la firma
Get-AuthenticodeSignature .\build\zentria-cli.exe
# Status: Valid
```

---

### ✦ 2. Certificado embebido como asset SEA  
**Commit:** `d060f25`

El archivo `ZentriaCertificado.cer` se incrusta dentro del propio `.exe` como un asset del blob SEA (junto a `bundle.mjs` y `yoga.wasm`).

**En `sea-config.json` (generado dinámicamente):**
```json
{
  "assets": {
    "ZentriaCertificado.cer": "build/ZentriaCertificado.cer"
  }
}
```

**En `sea-entry.cjs`:**
- Extrae el `.cer` a `%TEMP%` al iniciar.
- Establece `process.env.ZENTRIA_CER_PATH` apuntando al archivo temporal.
- La limpieza del archivo temporal se mueve de `try/finally` a `process.on('exit')` para que React pueda acceder al `.cer` durante el setup.

---

### ✦ 3. Instalación automática del certificado — `Bootstrapper`  
**Commits:** `d060f25`, `5adba84`

**`source/services/certificate.service.ts`** — Nuevo servicio con tres responsabilidades:
- `check()` — Verifica si el certificado `ZentriaCLI` está en `LocalMachine\Root` **Y** si ya hay una exclusión de Defender para el proceso. Usa PowerShell vía `execSync`.
- `install()` — Escribe un `.ps1` temporal y lo ejecuta con `Start-Process -Verb RunAs` (solicita UAC). El script:
  - Importa el `.cer` a `Cert:\LocalMachine\Root` (confianza global en el equipo).
  - Agrega `Add-MpPreference -ExclusionPath` para la carpeta del ejecutable.
  - Agrega `Add-MpPreference -ExclusionProcess "zentria-cli.exe"`.
- `isSetupDone()` / `markSetupDone()` — Flag persistido con **Conf** (`security_setup_done`) para no repetir la instalación.

**`source/components/Bootstrapper.tsx`** — Nuevo componente que actúa como **puerta inicial** antes de `LoginView`:
- En modo dev o si el flag local existe → salta instantáneamente.
- Si la verificación falla → muestra spinner animado y solicita certificado vía UAC.
- Al completar → llama `onReady()` y cede el paso al flujo normal.

**`source/app.tsx`** — Modificado para renderizar `<Bootstrapper onReady={...} />` antes de `LoginView`/`Shell`.

### ✦ 4. Bypass de SmartScreen — Distribución ZIP + `.bat`  
**Commits:** `80eb14d`, `de1030c`, `d0fe5a9`

> **Problema:** SmartScreen bloquea `.exe` desconocidos *antes* de que el código del ejecutable pueda ejecutarse — el certificado autofirmado por sí solo no es suficiente. Los archivos `.zip` y `.bat` no están sujetos a esta restricción.

**Solución adoptada:** El build genera un ZIP con 3 archivos:

```
Zentria-CLI-v1.1.6.zip
├── zentria-cli.exe          ← Ejecutable firmado
├── ZentriaCertificado.cer   ← Certificado público para instalar
└── Iniciar Zentria.bat      ← Launcher con setup de seguridad
```

**`Iniciar Zentria.bat`** (generado por `build-exe.mjs`):

| Paso | Acción |
|------|--------|
| 1 | `Unblock-File` sobre toda la carpeta (elimina Zone.Identifier / marca de internet) |
| 2 | Consulta `HKCU\Software\ZentriaCLI\SetupDone` para detectar primera ejecución |
| 3 | Si es primera vez: crea `.ps1` temporal y lo ejecuta elevado (UAC) → instala cert + exclusiones Defender |
| 4 | Escribe la clave de registro para evitar repetir el setup |
| 5 | `start "" zentria-cli.exe` — lanza el CLI |

### ✦ 5. Versionamiento del ZIP  
**Commit:** `14cb744`

El nombre del ZIP de distribución ahora incluye el número de versión tomado dinámicamente de `package.json`:

```js
// build-exe.mjs
const pkg = JSON.parse(readFileSync(join(ROOT, 'package.json'), 'utf-8'));
const VERSION = pkg.version;

const DIST_ZIP = join(BUILD_DIR, `Zentria-CLI-v${VERSION}.zip`);
```

Los metadatos del `.exe` (FileVersion, ProductVersion en resedit) también se sincronizan con `VERSION`.

**`package.json`** actualizado de `"0.0.0"` a `"1.0.0"` → genera `Zentria-CLI-v1.0.0.zip`.

### ✦ 6. GitHub Actions — Workflow de releases automáticos  
**Archivo:** `.github/workflows/release.yml`

Workflow que se activa al hacer push de un tag `v*`:

```yaml
on:
  push:
    tags:
      - 'v*'
```

**Pipeline en `windows-latest`:**
1. Checkout del código.
2. Setup Node.js 22.
3. `npm ci --legacy-peer-deps`.
4. `node scripts/build-exe.mjs`.
5. Publica el release en GitHub con el ZIP adjunto vía `softprops/action-gh-release@v2`.

## Flujo completo para el usuario final

```
1. Recibe Zentria-CLI-v1.1.6.zip por Teams/correo
2. Extrae el ZIP en una carpeta
3. Doble clic en "Iniciar Zentria.bat"
4. Primera ejecución: aparece UAC → "Sí" → instala certificado y exclusiones
5. El CLI se abre directamente (ventana de terminal con gradiente)
6. Próximas ejecuciones: arranca instantáneamente sin pasos de seguridad
```
